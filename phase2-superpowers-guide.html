Before You Start

이 가이드가 뭔가요?

Superpowers를 설치하고 실제로 기능 하나를 만들어본 전 과정이에요. 명령어만 나열하지 않고 "지금 뭐가 일어나고 있는 건지"를 함께 설명해요.

❌ Superpowers 없이

  • "카테고리 필터 만들어줘" 입력
  • Claude가 바로 코드 작성 시작
  • 앵커 링크가 가리킬 헤딩이 없어서 링크 깨짐 버그 발생
  • 나중에 발견하고 다시 수정

✅ Superpowers 있으니까

  • "카테고리 필터 만들어줘" 입력
  • brainstorming 먼저 — "헤딩이 없어서 앵커 링크가 안 됩니다"
  • 구조 설계 후 승인 → 구현
  • 버그 없이 첫 번째 시도에 완성
Superpowers의 핵심은 속도가 아니라 순서예요. 생각 → 설계 → 계획 → 구현 순서를 강제해서, 나중에 고치는 시간을 아껴줘요.
Overview

오늘 진행한 7단계

Superpowers는 항상 이 순서로 작동해요. 단계를 건너뛸 수 없어요.

Step 1–3
설치 + 활성화
Superpowers 플러그인 설치 → 적용 → 세션에 활성화. Claude Code에 "매뉴얼"을 심는 과정.
Step 4
Brainstorming
작업 요청 → Claude가 3번 질문으로 요구사항 확정. 바로 만들지 않고 "뭘 원하는지" 먼저 파악.
Step 5
설계 문서 + Git Commit
확정된 요구사항을 설계 문서로 작성 후 자동으로 git에 저장. "이렇게 만들 거야"의 공식 기록.
Step 6
구현 계획 (Writing Plans)
작업을 2~5분 단위 태스크로 분해해서 파일로 저장. 태스크마다 "어떤 파일을, 어떻게" 명시.
Step 7
서브에이전트 실행 + 검토
태스크마다 독립 에이전트가 구현 → Spec 검토 → 품질 검토. 총 6개 에이전트 투입 후 자동 git commit.
Step 1

wanna-try 폴더로 이동

Claude Code는 항상 "지금 어느 폴더에 있는지"를 기준으로 작동해요. 작업할 프로젝트 폴더로 먼저 이동해야 해요.

비개발자를 위한 설명: Claude Code는 폴더 단위로 작업해요. 작업하고 싶은 프로젝트 폴더로 "들어가는" 것이 첫 번째 단계예요. 마치 Finder(Mac) 또는 파일 탐색기(Windows)에서 폴더를 더블클릭해서 들어가는 것과 같아요.
Terminal 
# wanna-try 폴더로 이동
cd ~/Desktop/wanna-try

# Claude Code 실행
claude
💡 macOS 팁: 폴더 경로를 모르면 Finder에서 폴더를 찾아 터미널 창으로 드래그하면 경로가 자동으로 입력돼요.
PowerShell (LG Gram) 
# wanna-try 폴더로 이동 (kom-tank 기준)
cd ~\kom-tank\wanna-try

# Claude Code 실행
claude
🪟 Windows 팁: 경로를 모르면 파일 탐색기에서 폴더를 열고, 주소창 클릭 → 경로 복사 후 cd 경로 입력하면 돼요. 또는 폴더에서 Shift + 우클릭 → PowerShell 창 열기를 사용하면 바로 그 폴더에서 열려요.
Step 2

Superpowers 설치

Claude Code 안에서 플러그인을 설치해요. npm 같은 별도 설치 도구 없이 명령어 하나로 끝나요.

비개발자를 위한 설명: Superpowers는 Claude Code에 "업무 매뉴얼"을 심는 플러그인이에요. 설치하면 Claude가 바로 코드를 쓰지 않고 먼저 계획을 짜는 습관이 생겨요.

Step 3

Superpowers 활성화

설치만으로는 부족해요. 매 세션 시작 시 "이번 세션에 Superpowers 쓸 거야"라고 선언해야 스킬들이 자동 트리거돼요.

비개발자를 위한 설명: 플러그인을 설치한 건 "매뉴얼을 책상 위에 올려둔 것"이고, 활성화는 "오늘 이 매뉴얼대로 일할게"라고 선언하는 거예요. 이 선언 없이 작업 요청하면 Claude가 예전 방식대로 바로 코드를 써요.
Claude Code 프롬프트 (Mac/Windows 동일)
/using-superpowers
🪟 Windows 주의사항: Claude Code를 껐다가 다시 켜면 매번 /using-superpowers를 다시 입력해야 해요. 자동으로 유지되지 않아요. Phase 5에서 Session-start Hook을 설정하면 자동화할 수 있어요.

Step 4 — Brainstorming

작업 요청 → 질문으로 요구사항 확정

Superpowers가 활성화된 상태에서 작업을 요청하면, Claude가 바로 만들지 않고 먼저 질문을 해요. 이게 brainstorming 스킬이에요.

비개발자를 위한 설명: 건축가에게 "집 지어줘"라고 하면 바로 공사하지 않고 "방은 몇 개? 마당 필요해?" 먼저 물어보죠. Superpowers의 brainstorming이 딱 그 역할이에요. 질문 3번으로 "정확히 뭘 원하는지"를 확정해요.
이렇게 요청했어요
wanna-try 앱에 카테고리 필터 기능을 추가해줘

Claude의 3번 질문 흐름

Q1
카테고리 필터가 어떤 의미인가요?
a) 빠른 탐색 링크 b) 별도 섹션 c) 스크립트
왜 이 질문이 중요한가: "필터"라는 말의 해석이 사람마다 달라요. Claude가 임의로 해석하지 않고 먼저 확인하는 거예요.
내 답변
a
Q2
앵커 링크가 가리킬 곳이 없어요
기존 구조는 상태 기준 헤딩만 있어서 카테고리 앵커가 동작 안 함
여기가 핵심이에요! Superpowers 없었으면 Claude가 그냥 링크를 만들었다가 나중에 "링크가 안 돼요"라는 버그를 발견했을 거예요. brainstorming이 이걸 미리 잡아낸 거예요.
내 답변
a  (상단 링크 → 하단 카테고리 섹션으로 이동)

  • 어디서나 작동
접근 B
  • Dataview 동적 쿼리

Step 5 — 설계 문서

설계 확정 → 자동 Git Commit

요구사항이 확정되면 Claude가 설계 문서를 작성하고 자동으로 git에 저장해요. "이렇게 만들 거야"의 공식 기록이 남는 거예요.

비개발자를 위한 설명: 건축 허가서 같은 거예요. 착공 전에 "이 구조로 짓겠다"고 서류로 남기는 것처럼, Claude가 구현하기 전에 설계를 문서로 저장해요. 나중에 "왜 이렇게 만들었지?"를 추적할 수 있어요.
실제로 나온 결과 
⏺ Bash(git commit -m "docs: 카테고리 필터 설계 문서 추가")
  ⎿  [main 98b88f1] docs: 카테고리 필터 설계 문서 추가
      1 file changed, 36 insertions(+)
      create mode 100644 wanna-try/docs/superpowers/specs/
                    2026-06-12-category-filter-design.md

스펙 자체 검토 완료 — 플레이스홀더 없음, 내부 일관성 OK, 범위 적절.
자동으로 일어난 것들: 설계 문서 파일 생성 → 스스로 검토 ("플레이스홀더 없음, 일관성 OK") → git add → git commit. 명령어를 하나도 입력하지 않았어요.
📁
저장된 위치: docs/superpowers/specs/2026-06-12-category-filter-design.md
날짜가 파일명에 자동으로 포함돼요.
Step 6 — Writing Plans

구현 계획 → 태스크 분해

설계가 확정되면 Claude가 작업을 2~5분 단위로 잘게 쪼개서 계획 파일을 만들어요. 그리고 실행 방식을 선택하게 해요.

비개발자를 위한 설명: 요리 레시피처럼 "①양파를 썰고 ②불을 켜고 ③볶는다" 식으로 순서를 쪼개는 거예요. 태스크가 작을수록 에이전트가 실수할 확률이 낮아져요.
실제로 나온 결과 
⏺ Write(docs/superpowers/plans/2026-06-12-category-filter.md)
  ⎿  Wrote 117 lines to plans file

계획이 저장됐습니다. 실행 방식을 선택해 주세요:

  1. 인라인 실행 — 지금 이 세션에서 순서대로 실행
  2. 서브에이전트 — 태스크별 독립 에이전트 실행 후 검토
1번 인라인 실행
  • 빠르고 단순
  • 지금 세션에서 순서대로 처리
  • 간단한 작업에 적합
2번 서브에이전트 ← 오늘 선택
  • 태스크마다 독립 에이전트 투입

💡
brainstorming이 버그를 미리 잡았어요
앵커 링크 문제를 구현 전에 발견

Superpowers 없이 그냥 "카테고리 필터 만들어줘" 했으면 Claude가 바로 링크를 만들었다가, 가리킬 헤딩이 없어서 링크가 작동 안 하는 버그가 발생했을 거예요. brainstorming 두 번째 질문에서 이걸 미리 발견하고 구조를 바꿨어요.

📝
모든 과정이 파일로 남아요
설계 문서 + 구현 계획 + git 히스토리

"왜 이렇게 만들었지?"를 나중에 추적할 수 있어요. docs/superpowers/specs/에 설계 문서, docs/superpowers/plans/에 구현 계획이 날짜별로 저장돼요.

⏱
느린 것 같지만 결국 빠른 방식이에요
4분 27초 걸렸지만 수정이 없었어요

"바로 만들기" 방식은 빠른 것 같지만 수정이 필요할 때마다 시간이 추가돼요. Superpowers는 처음에 시간을 더 쓰는 대신 수정 없이 한 번에 완성이에요.
